# ADR 004: Подписки и фиксированные расходы

**Статус:** ПРИНЯТО
**Дата исходного решения:** 2024-05-29
**Дата актуализации:** 2026-06-28
**Источник актуализации:** `FamTrack@768af344db91`

## 1. Проблема

Регулярные расходы часто не видны в ежедневном учёте: сервисы, аренда, интернет,
обучение, подписки и автоплатежи. Без отдельной модели семья недооценивает
monthly burn rate.

## 2. Решение

Введена сущность `Subscription`, которая описывает шаблон регулярного платежа:

- название;
- сумма;
- валюта;
- service id;
- частота;
- следующая дата оплаты;
- auto-pay marker;
- счёт списания;
- категория;
- active flag.

Оплата подписки создаёт реальную `Transaction`, уменьшает баланс счёта,
пересчитывает `nextPaymentDate` и создаёт event `SUBSCRIPTION_PAID`.

## 3. Модель данных

Таблица `subscriptions` family-scoped и связана с `accounts` через `account_id`.
Фактическое списание фиксируется в `transactions`, а не только в самой подписке.

## 4. Процесс оплаты

1. Пользователь выбирает подписку.
2. Клиент проверяет наличие счёта и достаточность баланса для UX.
3. Клиент создаёт transaction payload.
4. Клиент обновляет account balance.
5. Клиент рассчитывает следующую дату через recurring helper.
6. Compound update отправляется в backend.
7. Backend проверяет роль и сохраняет данные транзакционно.

## 5. UX-требования

- Видеть ближайшие платежи.
- Отличать просроченные и upcoming payments.
- Показывать fixed monthly cost.
- Давать one-tap payment для ручных платежей.

## 6. Риски и развитие

- Нет фонового автоматического списания.
- Нет server-side command, который гарантирует атомарную бизнес-операцию
  подписки независимо от клиента.
- Для будущей версии полезен календарь платежей и notification layer.
