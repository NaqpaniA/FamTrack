# ADR 002: Совместные накопительные цели

**Статус:** ПРИНЯТО В ТЕКУЩЕЙ МОДЕЛИ
**Дата исходного решения:** 2024-05-25
**Дата актуализации:** 2026-06-28
**Источник актуализации:** `FamTrack@768af344db91`

## 1. Контекст

FamTrack закрывает короткие ежедневные сценарии через задачи и XP, но семье
также нужны среднесрочные цели: крупная покупка, поездка, подарок, техника или
обучение. Простого `Account` недостаточно, потому что детская цель может быть
не реальным банковским счётом, а семейной договорённостью.

## 2. Решение

В системе есть отдельная сущность `SavingsGoal`, независимая от legacy
`FinancialGoal`.

`SavingsGoal` хранит:

- название;
- описание;
- target amount;
- current amount;
- status;
- иконку;
- автора;
- дату создания.

`GoalContribution` хранит историю пополнений цели:

- goal id;
- user id;
- сумму;
- сообщение;
- дату.

## 3. Модель данных

Текущие таблицы:

- `savings_goals`
- `goal_contributions`
- `transactions`
- `accounts`
- `events`

При взносе в цель клиент формирует составной update:

1. Уменьшить баланс source account.
2. Увеличить `current_amount` цели.
3. Добавить `goal_contribution`.
4. Добавить `transaction` категории `goal_contrib`.
5. Добавить activity event `GOAL_CONTRIBUTION`.

Backend применяет update транзакционно через `FamTrackDatabase.mutate()`.

## 4. Авторизация

Finance operations доступны owner/admin. Child видит только разрешённую часть
финансовых данных и не должен обходить ограничения через batch update.

Серверные ограничения:

- `assertCanWrite()` проверяет route-level права.
- `sanitizeBatchUpdates()` ограничивает batch keys по роли.
- `filterForActor()` убирает невидимые счета и цели из read model.

## 5. UX-требования

- Цель должна показывать понятный прогресс.
- История пополнений должна объяснять, кто и когда внёс вклад.
- При успешном взносе activity feed должен дать семье контекст.
- Ошибки недостаточного баланса или некорректной суммы должны показываться до
  отправки, но backend всё равно остаётся enforcement layer.

## 6. Риски

- Текущая модель не разделяет физическое движение денег и семейную договорённость
  на уровне ledger subtype.
- Нет отдельного серверной команды `contributeToGoal`; составной update
  собирает клиент.
- Для будущей финансовой точности может понадобиться granular transaction API.
